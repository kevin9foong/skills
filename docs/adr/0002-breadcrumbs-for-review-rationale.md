# Rationale capture for PR review via breadcrumbs, not post-hoc derivation

The `prepare-for-review` skill needs to surface *why* code looks the way it does — alternatives considered, design trade-offs — so reviewers can evaluate decisions, not just read the diff. Two ways to source this rationale: (a) re-derive at PR-creation time by reading the diff + spec and reasoning about what alternatives existed, or (b) require implementation skills to drop **breadcrumbs** (small structured decision notes in `.scratch/<feature>/decisions.md`) as decisions happen, then aggregate at the end.

We picked (b). Re-derivation is fluent and convincing but fundamentally a hallucination: the author-agent makes up plausible alternatives it never actually weighed. Reviewers learn to trust the rationale; the trust is misplaced. Breadcrumbs are honest — the only rationale that surfaces is rationale that was real at the time. The cost is real: implementation skills (`tdd` and any future impl loops) must follow a breadcrumb convention defined in `docs/agents/decisions-breadcrumb.md` in the consuming repo, and `prepare-for-review` becomes a pure transport layer rather than a generator. Larger architectural decisions still go through ADRs via `grill-with-docs`; breadcrumbs cover the in-impl judgment calls that are too narrow to warrant an ADR.
